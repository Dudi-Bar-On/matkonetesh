## 0. Current state (verified in code)

- `probeChannels()` (`app.js:229`) only sums `cap.channels` across registered probe-category equipment: `equipByCat('probe').reduce(...)`. It renders one static footer chip (`app.js:6425`, `🎯 N probes · N channels`). No live data flows into it.
- `navigator.bluetooth` does not appear anywhere in `app.js` — zero Bluetooth code exists today.
- Equipment already models a `channels` capacity (`app.js:85`, `capKey:'channels'`) and an `accuracy` property (`app.js:88`, `bounds:[0.1,5]`, unit `±°C`) that nothing currently reads.
- The Live Cook Copilot's pace engine (`copilotPace`, `app.js:5395-5416`) already consumes a `session.probes[]` array of `{t, tempC}` readings to compute rate-of-rise, stall detection (flat 65–77°C), and an ETA/verdict against `serveTs`. Readings are appended today only by manual entry: `copilotLogProbe(tempC)` (`app.js:5381`) is wired to a number input + "Log" button in the probe check-in card (`app.js:5471-5484`, `#copProbe` / `#copProbeLog`).
- The `bcheck` stage kind (`app.js:3261`, `5159-5161`, `5815`, `5993`) is the safety gate — "internal temp check, target N°" — that today also relies on the user reading a physical thermometer and confirming manually.

This confirms the brief: the integration surface (data model + consumer logic) already exists; only the *acquisition* of readings is unbuilt.

## 1. Web Bluetooth reality

**Verdict: works today on Android Chrome/Edge/Samsung Internet and desktop Chromium; categorically absent on iOS/iPadOS in any browser, including PWA-installed mode.**

- Web Bluetooth (`navigator.bluetooth.requestDevice()` → GATT connect → characteristic read/notify) is shipped in Chromium-based browsers: Chrome 56+, Edge 79+, Opera 43+, Samsung Internet 6.2+, on Windows, macOS, Linux, ChromeOS, and Android. Global feature-support is ~76% as of April 2026, almost entirely because of the iOS/Safari gap. — [caniuse: Web Bluetooth](https://caniuse.com/web-bluetooth), [MDN Web Bluetooth API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Bluetooth_API)
- **Firefox and Safari (all platforms — macOS, iOS, iPadOS) do not implement Web Bluetooth at all**, and Apple has given no signal of intent to ship it. This is a hard platform wall, not a version gap. — [TestMu/LambdaTest: Web Bluetooth browser support](https://www.testmuai.com/learning-hub/web-bluetooth-browser-support/), [Chrome Community: Web Bluetooth on iOS](https://support.google.com/chrome/thread/33293081/use-web-bluetooth-api-with-google-chrome-on-ios?hl=en)
- A concrete positive proof-point that this is real, working technology (not theoretical): a public gist demonstrates `navigator.bluetooth.requestDevice()` connecting to an actual BBQ probe (Oregon Scientific "Grill-Right") over a custom GATT service and reading live temperature notifications, tested on Chrome for Android and macOS. — [BBQ Thermometer with the Web Bluetooth API (gist)](https://gist.github.com/mattdsteele/0fa9cafc4a95738181137547eae21fc8)
- **Workaround for iOS**: Bluefy — a third-party iOS browser app (App Store, iOS 11+) that polyfills `navigator.bluetooth` by bridging to Apple's native CoreBluetooth framework. A user would have to install this dedicated browser and open the PWA's URL inside it rather than use Safari or the home-screen-installed PWA. It is a real, shipping option but a meaningfully worse UX than "just works," and it's unclear whether an installed (add-to-homescreen) PWA can run inside Bluefy at all vs. only browsing to the URL. — [Bluefy — Web BLE Browser (App Store)](https://apps.apple.com/us/app/bluefy-web-ble-browser/id1492822055), [PNN Soft on Bluefy](https://pnnsoft.com/software-development-company/news/web-ble-for-ios-bluefy-browser-today-at-app-of-the-day)

**Backgrounding / screen-off — the second hard limitation, even on Android:**

- "A PWA that runs a Bluetooth-connected service in the background, like a native app, is currently not possible on any platform" (Chrome team framing). Bluetooth traffic is throttled once a tab is backgrounded and further throttled once the screen turns off; connections can silently drop. — [Progressier: Web Bluetooth API PWA capability](https://progressier.com/pwa-capabilities/bluetooth)
- On Android specifically, even a foreground service is not a Web Bluetooth concept — that's a native-app-only escape hatch. A web page's GATT connection is tied to the page's lifecycle; deep Doze can defer scans/drop connections when the screen is off and the phone is stationary. — [Android BLE Background Scanning Limits](https://www.devsflow.ca/blog/ble-android-lessons.html), [Android Developers: Communicate in background](https://developer.android.com/develop/connectivity/bluetooth/ble/background)
- The Screen Wake Lock API can keep the *screen* on (preventing the OS from sleeping the device), which is the practical mitigation for a live cook — but it drains battery, requires the tab to stay foregrounded/visible, and is released automatically the moment the tab is backgrounded, minimized, or the window loses focus (a `visibilitychange` handler can attempt to re-acquire it, but only once foregrounded again). — [Progressier: Screen Wake Lock PWA capability](https://progressier.com/pwa-capabilities/screen-wake-lock), [What PWA Can Do Today: Wake Lock](https://whatpwacando.today/wake-lock/)

**Bottom line for a 12+ hour cook (e.g., overnight brisket):** a Web Bluetooth connection realistically cannot be trusted to survive unattended for that duration with the phone locked/screen off. It's viable for a phone kept plugged in, screen-on or frequently reopened (e.g., propped on a counter as a dedicated "cook monitor"), but not as a fire-and-forget background service the way the vendors' own native apps are (native apps get OS-level Bluetooth background entitlements a web page cannot obtain).

## 3. Log-import fallback (the realistic iOS path)

For every vendor without an open BLE spec or official API, the pattern converges on the same shape: **the vendor's own app already renders a temperature-vs-time graph and offers a post-cook CSV export**, obtained by email/share-sheet from inside their app:

- **ThermoWorks Cloud**: web client (`cloud.thermoworks.com`) has an explicit "export to Excel 2007+ or .csv" menu option, plus PNG/JPG/PDF graph export. — [ThermoWorks Cloud](https://help.thermoworks.com/knowledge-base/thermoworks-cloud/)
- **Weber Connect app**: "graph data export... saves cooking data for in-depth analysis and exports to CSV." — Weber Connect app store listings
- **Govee Home app**: exports up to 2 years of history to CSV via email.
- **Inkbird (BBQ Go / Engbird)**: History tab → Export → CSV, per-probe (known limitation: can't export "all probes" in one action on some app versions).
- **Meater**: no confirmed native CSV export in-app, but "Previous Cooks" graphs are viewable in-app, and the beta Cloud REST API can programmatically retrieve prior cook data (a GitHub discussion specifically covers pulling previous-cook info via the API) — so import doesn't strictly need a file, it could poll the same JSON the app uses. — [MEATER API — previous cook discussion](https://github.com/apption-labs/meater-cloud-public-rest-api/discussions/34)

**Implication for matconetesh:** a "log-import" feature (user exports CSV from their probe's app after a cook, uploads/pastes it into the PWA) is buildable *today*, works identically on iOS and Android since it's just a file, and needs no Bluetooth/API integration at all. Its value is retrospective (verify the plan after the fact, build a temperature graph, catch "did I actually hit safe temp") rather than live — it cannot feed the Copilot's live pace/ETA/stall detection, which needs readings *during* the cook.

## 4. Sous-vide circulators (Anova / Joule)

- **Anova**: as above — WiFi-connected Anova devices (Precision Cooker 3.0/A3, Precision Oven) expose bath temperature and setpoint through the **official cloud API**, which is reachable from a browser via plain HTTPS (subject to CORS — unverified whether Anova sets permissive CORS headers for arbitrary web origins; if not, a small server-side proxy closes the gap). BLE-only models (Mini, Nano) need Web Bluetooth, so Android-only, foreground-only. — [Anova Developer Docs](https://developer.anovaculinary.com/), [Anova Nano device docs](https://developer.anovaculinary.com/docs/devices/nano/overview)
- **Joule (ChefSteps → now Breville+ app)**: has both Bluetooth and WiFi; Bluetooth is used to *start* a cook, but "once the cook has started, Joule will continue maintaining temperature even if you leave Bluetooth range," implying it relies on Wi-Fi/cloud for anything beyond the initial pairing. **No public/official API found** for Joule — support docs only, no developer portal. A community iOS wrapper (`brennanangel/circulate`) reverse-engineers Anova's *older* v2 Bluetooth protocol, not Joule. — [ChefSteps Joule Connectivity](https://support.chefsteps.com/hc/en-us/sections/205091008-Joule-Connectivity), [circulate (community, Anova v2)](https://github.com/brennanangel/circulate)

**Verdict: remote read of bath temperature/setpoint from a browser is feasible today for Anova's WiFi-connected models via the official cloud API — this is the single most "just works, cross-platform, no BLE needed" integration in this whole survey.** Joule has no such opening.

## 5. What it would unlock in this app (concrete)

1. **Automatic Live Cook Copilot readings.** `copilotLogProbe(tempC)` (`app.js:5381`) is the exact function a background poller or BLE notification handler would call instead of the manual `#copProbeLog` button. `copilotPace()` (`app.js:5395`) already computes rate-of-rise, stall (flat 65-77°C), and ETA/verdict from whatever lands in `session.probes[]` — it is *agnostic* to how readings arrive. Wiring one data source (start with Anova cloud, since it needs no Bluetooth) would upgrade the Copilot from "you type in a number every so often" to "it just knows," with zero changes to the pace math.
2. **`bcheck` safety gate becomes machine-verifiable.** Today `bcheck` stages (`app.js:3261`, `5159`) tell the user "check with a thermometer" and trust manual confirmation. A live feed lets the app auto-detect when target is crossed and surface it as a real-time alert rather than a static instruction — directly serving the existing `safe` internal-temp targets already in the recipe model.
3. **Probe-channel budgeting becomes real.** `probeChannels()` (`app.js:229`) currently just displays an inert count. With live device data, the app could actually check "this cook plan calls for monitoring 3 cuts simultaneously; your registered gear (`cap.channels` across probe-category equipment) only covers 2 channels" — turning the already-modeled `channels`/`accuracy` equipment properties from decorative into functional, and flagging a real gear gap before the cook starts, not during it.
4. **Sous-vide bath verification** — for Anova WiFi users, the app could show live bath temp vs. the recipe's specified setpoint, catching a bath that never reached temperature (a real food-safety failure mode for long sous-vide cooks) without the user needing to check a separate app.

## 6. Recommendation — staged path, cheapest/most-certain first

1. **Log-import (all platforms, ship first).** Accept a pasted/uploaded CSV (ThermoWorks, Weber Connect, Govee, Inkbird all export it natively) and render a post-cook temperature graph against the plan's `safe` target and `bcheck` stages. Zero new permissions, zero Bluetooth, works identically on iOS and Android. This alone converts "no probe integration" into "verify your cook after the fact" — the single highest-certainty, lowest-risk increment.
2. **Anova cloud API (WiFi models), Android + iOS via HTTPS.** Poll the official Developer API for live bath temperature/setpoint during a sous-vide cook and feed it straight into the existing pace/`bcheck` UI. Needs a CORS check (likely a thin proxy) but no native app, no Bluetooth — the best risk/reward integration found in this whole survey, and it lands on iOS too.
3. **MEATER Cloud REST API, same shape as #2 but beta/unofficial-adjacent** (Apption Labs' own repo calls it BETA) — reachable from any browser via HTTPS if the user has the MEATER app or Block bridging to their cloud account; same CORS caveat.
4. **Web Bluetooth live readings, Android/desktop Chrome only, foreground-biased.** Combustion Inc's open MIT-licensed GATT spec is the one vendor where building a compliant client from official docs (not reverse-engineering) is realistic; feed `copilotLogProbe()` directly. Explicitly scope this as "works best with the phone propped up and the screen on" — do not promise unattended 12-hour background monitoring; the platform genuinely cannot deliver that from a web page.

**Honest blockers to name up front:** (a) iOS/Safari has zero Web Bluetooth support and no roadmap signal from Apple — any live-BLE feature is Android/desktop-only unless the user installs a niche third-party browser (Bluefy); (b) even on Android, a backgrounded/screen-off tab cannot reliably hold a GATT connection for a 12+ hour cook — this is a page-lifecycle limitation, not a bug to be fixed later; (c) most vendors' cloud APIs are undocumented for CORS behavior — assume a small server-side proxy is needed until verified otherwise.
