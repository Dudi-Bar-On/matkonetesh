# i18n / Multi-language Audit — 2026-07-15

*Read-only audit of the translation machinery before scaling from he/en to he/en/fr/de/es. Verdict: the mechanism needs a small, high-leverage improvement before adding languages — not a rewrite.*

## What's good (keep it)
- **The `tnode()` + per-language dict layer is genuinely solid and scalable.** Dicts are `lang/<code>.json` (chrome) + `lang/<code>.data.json` (prose), keyed by **the literal Hebrew source string → translation**. `build.py` (~341-357) globs `lang/*.json`, merges `.data.json` into the base, inlines as `I18N_DICTS`. **Adding a language file is zero-code** — drop `lang/it.json` and it auto-appears in the switcher (`I18N_LANGS`, `langRowHtml`). `getLang()` is host-pluggable (`__MATKONET_HOST__.lang` → `mk-lang` → 'he'); `restoreHe()`/`_mkO` make switching back to Hebrew instant + lossless. RTL/LTR from each dict's `__meta__.dir` — fr/de/es are all LTR so they inherit English's solved path.

## The ceiling (the real problem)
- **`L(he, en)` (app.js:5086) is a hard 2-language switch** — `getLang()==='he' ? he : en`. It does **not** consult the dict. Used at **~1,333 call sites** in app.js — the single largest translation path, dwarfing the 305-key chrome dict. For fr/de/es, all 1,333 fall back to **English**. No discipline fixes this; each site needs the mechanism changed.
- **~10 duplicate binary helpers** hand-roll the same `getLang()==='he'?X:Y`: `itemName` (5080), `kindLabel` (346), `heatLabel` (371), `servTypeName/Note` (1111-2), `svOrderName/Desc` (2118-9), `themeName/fontName/scaleLabel` (5052-4), `uiLevelName/Desc` (5202-3), `shapeName` (5207) — each its own 2-language ceiling.
- **Catalog data is 2-field** (`heb`/`eng` per row; `itemName` collapses non-Hebrew→English). No fr/de/es name field in the data model.

## Present-day bugs the audit surfaced (broken NOW, not just future)
- **`aiJSON` `outLang` is broken for non-Hebrew** (app.js:3249-3250): any non-'he' outLang — including 'fr'/'de'/'es' — emits the hardcoded instruction "write every value in **ENGLISH**." So every AI feature is English-only for non-Hebrew, regardless of UI language.
- **`vcBuildAskPrompt` (voice)** (4104-4120): only `if ansLang==='en' … else Hebrew` — a non-he/en answer-lang falls into the **Hebrew** branch.
- **~20 hardcoded Hebrew `toast()` strings bypass i18n entirely** — not even in `en.json`, so they're **permanently Hebrew even in English mode**: storage-full (668), tiבול updated (1368), add-to-menu (2360), finishing-step (2381), undo toasts (2450/2479/2554), list-copied (3591), speech status/errors (4158-4194), backup/restore errors (4824-4837). ⇒ **English is NOT actually 100%** — the earlier Playwright sweep missed these because they're conditional/hidden (same class as the v189 banner lesson).
- **Generation-time `L()` headers outside `data-mt`** (e.g. app.js:1185 "Materials & equipment") show English next to fr/de/es-or-Hebrew `data-mt` list items → three-language mix on one screen for fr users.
- **No coverage/lint tooling**: rewording any Hebrew UI string silently orphans its dict key in all languages — no build warning, no test failure, string quietly reverts to Hebrew.
- **Dead `data-i18n="..."` markup** (build.py:160-243, ~14 els) is never read (only `data-i18n-html` is) — misleading remnants of an abandoned keyed-i18n attempt.

## English privilege
English is the de facto pivot everywhere outside the dict layer: `L()`'s only non-Hebrew output is English; the binary helpers collapse "not Hebrew"→English; `aiJSON` hardcodes ENGLISH; `vcLang` defaults non-he→'en'; `vcTranslateToEn` is English-specific. Only `tnode()`/dict and `mtTranslate()`'s `LANGNAME` map ({en,ar,ru,es,fr,de}, app.js:5156) are truly language-parametric.

## Recommendation (minimal, high-leverage — NOT a rewrite)
Priority order:
1. **Unify `L()` with the dict.** `L(he,en)` → `getDict()[he] ?? en ?? he`. **Signature-compatible**, non-breaking (English still works via the inline `en` arg where no dict entry exists), and makes **all ~1,333 sites multi-language-capable with no per-site change** — add the Hebrew key to `fr/de/es.json` and it localizes. *Single highest-leverage fix.*
2. **Fix `aiJSON` `outLang`** (3249-3250) to interpolate the real target language name (reuse the `LANGNAME` map). Small fix, fixes every AI feature (incl. the planned Equipment AI-spec fetch).
3. **Route the ~20 hardcoded Hebrew `toast()`s** through the dict — fixes a real *English* bug today.
4. **Add a build-time coverage check** to `build.py`: diff each `lang/*.json` against the used-Hebrew set / against `en.json`; print coverage % + stale keys. Only way to know what "add French" costs + catches silent orphaning.
5. **Collapse the ~10 binary helpers** onto `getDict()` lookup.
6. **Remove/wire the dead `data-i18n` markup.**

Items **1–2 alone** flip "does this scale?" from *no (a he/en switch with a chrome-only escape hatch)* to *yes (the same tnode/dict layer covers the whole app)*. A full semantic-keyed i18n rewrite would be more robust long-term (kills the "reword-Hebrew-silently-orphans" fragility) but is larger/riskier and not required to unblock fr/de/es.

## Implication for Equipment 2.0
The spec currently says "all new strings via `L(he,en)`." After fix #1, that becomes correct and future-proof (L is dict-aware). So: **do the i18n foundation (at least #1–2) first**, then Equipment 2.0 authors via the now-dict-aware `L()` + `data-mt` for prose + fixed `outLang` for its AI helper — born multi-language-ready with no extra authoring effort, and adds no new 2-language debt.
