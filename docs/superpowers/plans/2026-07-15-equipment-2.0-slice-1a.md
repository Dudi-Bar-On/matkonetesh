# Equipment 2.0 — Slice 1A (model + migration + port + manager) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the flat single-device `mk-gear` model with a multi-device `mk-equipment` list (foundation for the Cookout Orchestrator), port every consumer, add a manual equipment manager, and delete the old gear code — with the app fully working at every commit.

**Architecture:** `mk-equipment` = an array of device objects becomes the single source of truth. The capability *functions* (`canSV/canSmoke/canGrill/homeGear/smokerTip/preheatHint/gearThermoNote`) keep their names but are reimplemented to read the device list, so the ~dozen UI sites that call them need no change; only the ~6 sites that read `gearState()` directly are edited. A one-time `equipMigrateFromGear()` seeds the list from any existing `mk-gear`. `openGear` (dropdowns) is replaced by `openEquipment` (device manager). This is Slice 1A of Phase 1; the **AI equipment helper** and **item→cooker assignment** are separate follow-on slices.

**Tech Stack:** Vanilla browser JS (`app.js`), Python build (`build.py`), Playwright (`tests/*.spec.ts`).

## Global Constraints
- **No backward compat** (pre-customer): after migration, `mk-gear`/`gearState`/`saveGear`/`openGear`/`GEAR_GROUPS`-as-editor are **deleted**; consumers are ported. A one-time data migration preserves the owner's current kit.
- **All new UI strings via the now-dict-aware `L(he,en)`** (fr/de/es-ready by construction). Verify **Hebrew AND English**.
- **Reuse the existing Hebrew type vocabulary** as the device `type` values (e.g. `'אופסט / סטיק-ברנר'`, `'פרוב אלחוטי'`) so `SMOKER_TIPS`/`preheatHint`/`gearThermoNote` keep working unchanged.
- Ship as **v230** (bump `build.py:323` stamp, commit + tag + push). `node --check app.js` clean; full suite green **twice**.
- `L('…')` gotchas: no apostrophe inside single-quoted `L('…')`; no literal `״` in an Edit `old_string`.

## Device model (reference — implemented in Task 1)
```js
{ id, cat, type, name, brand, model, fuel, cap:{racks,zones,channels,bathL}, specSource:'manual'|'ai', notes }
```
`cat ∈ smoker|grill|oven|sousvide|probe|grinder|stuffer|other`. Capacity field by cat: smoker/oven→`racks`, grill→`zones`, probe→`channels`, sousvide→`bathL`; others none. Old `thermo`→`probe`; old extras (vacuum/injector/slicer/curechamber/humidity/scale/hooks/torch)→`other` with `type`=the old field id.
